                        9-STEP CHECKOUT WORKFLOW
═══════════════════════════════════════════════════════════════════════════════

                           HTTP REQUEST ARRIVES
                                  │
                                  ▼
                    ┌─────────────────────────────┐
                    │  OrderController validates  │
                    │  all required fields        │
                    └─────────────┬───────────────┘
                                  │
                                  ▼
                    ┌─────────────────────────────┐
                    │ Call ProcessCheckoutAsync() │
                    └─────────────┬───────────────┘
                                  │
        ┌─────────────────────────┘
        │
        ▼
    ┏━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┓
    ┃                                                       ┃
    ┃  ├─ STEP 1: BEGIN TRANSACTION                        ┃
    ┃  │           await _unitOfWork.BeginTransactionAsync()
    ┃  │           Opens database transaction               ┃
    ┃  │           All operations now atomic               ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┃  ├─ STEP 2: IDEMPOTENCY CHECK                        ┃
    ┃  │           SELECT * FROM Orders WHERE Number = ?  ┃
    ┃  │           IF EXISTS: Return cached order          ┃
    ┃  │           IF NOT: Continue to Step 3              ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┃  ├─ STEP 3: AUTHENTICATE USER                        ┃
    ┃  │           IF isAuthenticated:                      ┃
    ┃  │             → Get user by email                    ┃
    ┃  │           ELSE:                                    ┃
    ┃  │             → Call AuthService.Login()             ┃
    ┃  │             → On error: ROLLBACK (→ Error)        ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┃  ├─ STEP 4: MERGE GUEST CART                         ┃
    ┃  │           IF guestId provided:                     ┃
    ┃  │             → Get all GuestCart items              ┃
    ┃  │             → For each item:                       ┃
    ┃  │               - Check if in UserCart               ┃
    ┃  │               - Merge quantities or create new     ┃
    ┃  │             → Delete all GuestCart items           ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┃  ├─ STEP 5: CREATE ORDER                             ┃
    ┃  │           INSERT INTO Orders (UserId, Number, ...) ┃
    ┃  │           Status: OrderStatus.Pending              ┃
    ┃  │           TotalPrice: (to be calculated)           ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┃  ├─ STEP 6: RESERVE VARIANTS (Critical!)             ┃
    ┃  │           FOR EACH CartItem:                       ┃
    ┃  │             → Check: Variant.Quantity >= Qty?      ┃
    ┃  │             → IF NO: ROLLBACK (→ Error)           ┃
    ┃  │             → IF YES:                              ┃
    ┃  │               - Reduce: Variant.Quantity -= Qty     ┃
    ┃  │               - Reserve: Variant.Reserved += Qty    ┃
    ┃  │               - UPDATE Variant                      ┃
    ┃  │               - INSERT OrderDetail                  ┃
    ┃  │               - Add to TotalPrice                   ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┃  ├─ STEP 7: SAVE INVENTORY TRANSACTIONS               ┃
    ┃  │           FOR EACH OrderDetail:                    ┃
    ┃  │             → INSERT InventoryTransaction          ┃
    ┃  │             → TransactionType: Sale (enum)         ┃
    ┃  │             → Links Order, User, Variant           ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┃  ├─ STEP 8: CLEAR USER CART                          ┃
    ┃  │           DELETE FROM UserCarts WHERE UserId = ?   ┃
    ┃  │           Removes all processed items              ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┃  ├─ STEP 9: COMMIT TRANSACTION                        ┃
    ┃  │           SaveChanges()  → Persist to DB           ┃
    ┃  │           CommitAsync()  → Make permanent          ┃
    ┃  │           All changes now visible to others        ┃
    ┃  │                                                    ┃
    ┃  ▼                                                    ┃
    ┗━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━┛
                                  │
                  ┌───────────────┤
                  │               │
                  ▼               ▼
        SUCCESS RESPONSE    ERROR RESPONSE
        ┌──────────────┐    ┌──────────────┐
        │  200 OK      │    │  400/500     │
        │  {data}      │    │  {error}     │
        │  OrderId     │    │  Message     │
        │  OrderNumber │    │  Rollback ✓  │
        │  TotalPrice  │    │  Clean State │
        │  CreatedAt   │    └──────────────┘
        └──────────────┘


═══════════════════════════════════════════════════════════════════════════════
                          ERROR HANDLING & ROLLBACK
═══════════════════════════════════════════════════════════════════════════════

ERROR can occur at:
  • Step 2: Duplicate order (but returns success)
  • Step 3: User not found / Invalid credentials
  • Step 4: Cart merge fails
  • Step 5: Order creation fails
  • Step 6: Out of stock / Variant not found
  • Step 7: Transaction record fails
  • Step 8: Cart clearing fails
  • Step 9: Commit fails

ON ANY ERROR:
  ┌─────────────────────────────────────────┐
  │ catch (Exception ex)                    │
  │ {                                       │
  │   try                                   │
  │   {                                     │
  │     await _unitOfWork.RollbackAsync();  │
  │     ✓ Order: NOT created                │
  │     ✓ OrderDetails: NOT created         │
  │     ✓ Variants: NOT updated             │
  │     ✓ Cart: NOT cleared                 │
  │     ✓ InventoryTransactions: NOT added  │
  │     → User can RETRY with same key      │
  │   }                                     │
  │   catch (Exception rollbackEx)          │
  │   {                                     │
  │     Log("Rollback failed - CRITICAL!")  │
  │     Return error to client              │
  │   }                                     │
  │                                         │
  │   return Error(ex.Message);             │
  │ }                                       │
  └─────────────────────────────────────────┘


═══════════════════════════════════════════════════════════════════════════════
                          IDEMPOTENCY MECHANISM
═══════════════════════════════════════════════════════════════════════════════

REQUEST #1 (Original)
  ├─ IdempotencyKey: "550e8400-e29b-41d4"
  └─ Network timeout before response received

REQUEST #2 (Retry - Same IdempotencyKey)
  ├─ IdempotencyKey: "550e8400-e29b-41d4"
  ├─ Step 2: Check ORDER.Number = "550e8400-e29b-41d4"
  ├─ FOUND! (from Request #1)
  └─ Return: OrderId=42, OrderNumber="ORD-550e8400-e29b-41d4"

REQUEST #3 (Second Retry - Same IdempotencyKey)
  ├─ IdempotencyKey: "550e8400-e29b-41d4"
  ├─ Step 2: Check ORDER.Number = "550e8400-e29b-41d4"
  ├─ FOUND! (still Order #42)
  └─ Return: SAME ORDER (No duplicate!)

RESULT: ✓ Only ONE order created despite THREE requests
        ✓ Safe for network retries
        ✓ No duplicate orders
        ✓ Same OrderId returned


═══════════════════════════════════════════════════════════════════════════════
                        DATABASE STATE CHANGES
═══════════════════════════════════════════════════════════════════════════════

BEFORE CHECKOUT
  Orders Table:           (empty for this user)
  OrderDetails Table:     (empty)
  Variants Table:         Quantity=100, Reserved=0
  UserCarts Table:        [Item1, Item2, Item3] (user's cart)
  InventoryTransactions:  (no sale transactions)

DURING CHECKOUT (Inside Transaction)
  STEP 1: Transaction starts
  STEP 2-8: Changes made but NOT visible to others yet
  STEP 9: CommitAsync() → Changes become permanent

AFTER SUCCESSFUL CHECKOUT
  Orders Table:           [New Order with Pending status]
  OrderDetails Table:     [3 OrderDetail records]
  Variants Table:         Quantity=95, Reserved=5
                          (100 - 5 items ordered)
  UserCarts Table:        (EMPTY - cleared)
  InventoryTransactions:  [3 Sale transactions]

IF ERROR OCCURS
  Orders Table:           (No record added - Rollback worked)
  OrderDetails Table:     (Empty - Rollback worked)
  Variants Table:         Quantity=100, Reserved=0
                          (Unchanged - Rollback worked)
  UserCarts Table:        [Item1, Item2, Item3] (Unchanged)
  InventoryTransactions:  (No records - Rollback worked)
  → User can retry safely

═══════════════════════════════════════════════════════════════════════════════
                          PERFORMANCE TIMELINE
═══════════════════════════════════════════════════════════════════════════════

Timeline for a typical 5-item checkout:

Time (ms)    Event
─────────────────────────────────────────────────────────────
0            Request arrives
5            Request validation complete
10           ProcessCheckoutAsync() called
15           BEGIN TRANSACTION
25           Query: Check existing order (idempotency)
35           Query: Get user by email
45           Query: Get guest cart items (if guestId)
55           Merge guest cart (5 UPDATE + 5 DELETE)
100          INSERT Order
110          Query: Get user cart items with variants (5 items)
130          For each item: Stock check → UPDATE Variant (5 UPDATE)
180          INSERT 5 InventoryTransaction records
220          DELETE 5 UserCart items
250          SaveChanges() executed
260          CommitAsync() executed
270          Response prepared (CheckoutResponseDto)
280          Response serialized to JSON
290          Response sent to client

TOTAL: ~290ms (P95: 300-500ms under load)

═══════════════════════════════════════════════════════════════════════════════
                       INVENTORY TRANSACTION TYPES
═══════════════════════════════════════════════════════════════════════════════

InventoryTransactionType.Sale ◄── Used in checkout
  ├─ Records: Items sold to customer
  ├─ Quantity: Decreases (negative impact on stock)
  ├─ Impact: Variant.Quantity reduces
  └─ Example: -5 units (5 items sold)

Other types available for future use:
  ├─ Purchase: +50 units (restock from supplier)
  ├─ Refund: +5 units (customer returns items)
  ├─ Adjustment: ±? units (manual correction)
  ├─ Transfer: Move between warehouses
  └─ Reservation: Temporary hold (not yet sold)
